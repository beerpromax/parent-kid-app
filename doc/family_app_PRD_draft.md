## 一、产品定位

### 核心理念
一款连接家长和孩子的家庭协作移动应用，通过**代币奖励机制**将学习和活动游戏化。核心价值观：

- **平等** — 孩子和家长是伙伴关系，不是管控关系
- **共同成长** — 双方都在这个过程中获益
- **双赢** — 家长达成教育目标，孩子获得想要的奖励
- **快乐长大** — 过程本身是有趣的

## 二、核心逻辑

### 逻辑一：活动-奖励-代币系统

**基本流程：**
1. 家长创建活动列表（学习/练习目标），设定代币价值
2. 孩子创建奖励愿望清单，设定期望奖励
3. 双方协商达成"交易"——某活动对应某奖励，代币作为中间货币
4. 孩子完成活动 → 提交申请 → 家长审批 → 代币自动到账
5. 代币积累到足够数量 → 孩子兑换奖励

**代币属性：**
- 赚取：活动完成并审批后获得
- 价值：由家长设定，可协商调整
- 消费：兑换奖励愿望清单中的项目
- 连续打卡奖励（streak bonus）：连续完成活动可获得额外代币

### 逻辑二：成长日志

为每个孩子生成一份成长记录，当前版本追踪以下信息：
- 日期
- 参与者（谁和谁一起）
- 活动内容
- 持续时长
- 成就/完成情况
- 心情/能量标签（趣味评分）
- 短文字备注
- 照片附件

> 视频支持暂不在当前版本范围内

## 三、代币协商机制（详细规则）

这是整个app最核心、最有差异化的交互设计。

### 3.1 协商触发条件

| 条件 | 结果 |
|------|------|
| 该项目有待审批记录 | ❌ 不能发起协商申请 |
| 该项目无待审批记录 | ✅ 孩子可发起协商申请 |

**只有孩子能发起调整申请**（家长不能主动发起）

### 3.2 协商对象

两类项目均可发起协商：
- **活动的代币价值**（例如：读书30分钟 = 5个代币 → 申请改为8个）
- **奖励的代币成本**（例如：玩游戏1小时 = 20个代币 → 申请改为15个）

### 3.3 协商流程

```
孩子发起申请
    ↓
该项目进入"协商中"状态并锁定（不可再次发起新申请）
    ↓
家长收到通知，三个选项：
    ├── 直接同意 → 新汇率立即生效
    ├── 提出反价 → 进入来回协商
    └── 拒绝 → 维持原汇率，项目解锁
    ↓
（如进入协商）双方来回出价，次数不限
    ↓
双方都点"同意" → 新汇率生效
```

### 3.4 协商期间的并行规则

- 协商中的项目：活动照常可完成和提交审批
- 审批照常进行，**使用旧汇率**
- 协商和审批是**两条完全独立的并行轨道**，互不影响
- 新汇率只在双方达成一致后才对后续记录生效

### 3.5 历史汇率处理

协商达成前已完成但未审批的活动 → 按**旧汇率**结算，不受新协商结果影响。

## 四、用户角色

### 家长端
- 创建和管理活动库（活动名称、所需时长、代币价值）
- 管理交易/Deal（审阅孩子的奖励请求，设定代币成本）
- 一键审批孩子提交的活动完成申请
- 家庭总览仪表板（每个孩子的进度和代币余额）
- 处理协商请求（同意/还价/拒绝）

### 孩子端
- 查看今日活动列表（清晰显示每项活动的代币奖励）
- 代币钱包（余额 + 距离下一个奖励的进度条）
- 奖励愿望清单（可提议新奖励）
- 发起协商申请
- 奖励兑换流程（附庆祝动效）

## 五、数据结构（初步）

```
家庭账户
  └── 多个孩子账户

活动 (Activity)
  - 名称
  - 所需时长
  - 代币价值（当前生效版本）
  - 状态（正常 / 协商中）

完成记录 (CompletionRecord)
  - 关联活动
  - 关联孩子
  - 完成时间
  - 状态（待审批 / 已审批）
  - 结算代币数（按审批时的汇率）

奖励 (Reward)
  - 名称
  - 代币成本（当前生效版本）
  - 状态（正常 / 协商中）
  - 提出者（孩子）

协商记录 (NegotiationThread)
  - 关联项目（活动或奖励）
  - 发起方（孩子）
  - 消息列表（出价来回）
  - 状态（进行中 / 已达成 / 已拒绝）

成长日志 (GrowthLog)
  - 日期
  - 关联孩子
  - 活动内容
  - 时长
  - 参与者
  - 心情标签
  - 备注
  - 照片

代币钱包 (TokenWallet)
  - 关联孩子
  - 当前余额
  - 历史流水
```

## 六、技术方向

### 开发策略
**先做webapp验证核心逻辑和UI，再移植到移动端。**

原因：
- 原型迭代快，改动成本低
- 电脑端调试方便
- 逻辑跑通后，移植到React Native更顺滑

### 推荐技术栈
- **前端**：React（webapp阶段）→ React Native（移动端阶段）
- **后端/数据库**：Firebase 或 Supabase（待定，需进一步了解后决策）
  - Firebase 实时数据库天然适合代币变动这类实时场景
  - Supabase 是开源替代，结构更接近传统关系型数据库

### 开发阶段规划

| 阶段 | 内容 | 优先级 |
|------|------|--------|
| 第一阶段 | 核心交互环路：创建活动 → 完成提交 → 审批 → 代币到账 | 最高 |
| 第二阶段 | 代币系统：钱包、进度条、奖励兑换、协商机制 | 高 |
| 第三阶段 | 成长日志：时间线、照片、心情标签 | 中 |
| 移动端 | React Native移植 | 后续 |
